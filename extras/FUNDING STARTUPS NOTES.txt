The settings.py file
~~~~~~~~~~~~~~~~~~~~~
~~~~~~~~~~~~~~~~~~~~~
-> Why single settings.py is dangerous?

config/settings.py

Everything is inside ONE file:
debug
database
security
email
static files


Production risks:
debug: 	DEBUG = True (sensitive data leak)
database: Use MySQL or MS SQL Server or PostgreSQL
security: No HTTPS, weak secrets
secrets: API keys exposed in code


Industry Standard (solution)
----------------------------
config/
	settings/
		__init__.py
		base.py
		dev.py
		prod.py


Implementation
--------------
Step 1:

Creating 'settings' folder under config/

> mkdir config/settings

Create a file under 'settings' folder named __init__.py


Step 2:

Move settings.py -> base.py

config/settings.py	->  config/settings/base.py


Step 3:

Create dev.py and prod.py

config/settings/dev.py
config/settings/prod.py

Managing Secrets with .env (CRITICAL)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
- Stop hardcoding secrets in code
- Use .env file safely
- Load secrets into Django settings
- Prevent accidental leaks to GitHub


The problem (Very Dangerous)
-----------------------------

SECRET_KEY = 'django-insecure-e(w*8c30!g86-xh@z^iqwk_9^vb5efzg)cn_g=9uu#5e+@pwkp'

DATABASES = {
	'default': {
		'ENGINE': 'Django.db.backends.postgresql',
		'NAME': 'prod_db',
		'USER': 'postgres',
		'PASSWORD': 'secure_password',
		'HOST': 'localhost',
		'PORT': '5432'
	}
}



The Solution - .env file
-----------------------

Move the secrets into a separate file:

Environment:

SECRET_KEY = 'django-insecure-xyz123'
DEBUG = True
DB_NAME = mydb
DB_USER = postgres
DB_PASSWORD = supersecret

Common Practices
~~~~~~~~~~~~~~~~
~~~~~~~~~~~~~~~~
1. .env not in root

Correct location:
	manage.py
	.env

2. Quotes in .env

Wrong:
	SECRET_KEY = 'abc123'

Right:
	SECRET_KEY = abc123


3. Forgetting cast=bool

Wrong:
	DEBUG = config('DEBUG')

Right:
	DEBUG = config('DEBUG', cast=bool)



Security Upgrade
~~~~~~~~~~~~~~~~
~~~~~~~~~~~~~~~~

Before:
	SECRET_KEY = 'hardcoded'

Now:
	SECRET_KEY = config('SECRET_KEY')

Means:
	- Different key in dev vs prod
	- No exposure in code
	- Safe deployment

Auto Switch Between 'dev' and 'prob' Settings
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Problem
~~~~~~~~

Before:
	python manage.py runserver --settings=config.settings.dev

-> manual switching
-> deployment risk
-> team confusion


Solution - Environment Variable Switch
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

We will control everything using:

DJANGO_SETTINGS_MODULE


Update manage.py
~~~~~~~~~~~~~~~~

Open manage.py and replace:

os.environ.setdefault(
	'DJANGO_SETTINGS_MODULE',
	'config.settings'
)


With this:

import os

ENVIRONMENT = os.getenv('DJANGO_ENV', 'dev')

if ENVIRONMENT == 'prod':
	os.environ.setdefault(
		'DJANGO_SETTINGS_MODULE',
		'config.settings.prod'
	)
else:
	os.environ.setdefault(
		'DJANGO_SETTINGS_MODULE',
		'config.settings.dev'
	)




Update wsgi.py and asgi.py
~~~~~~~~~~~~~~~~~~~~~~~~~
Do the SAME logic:

import os

ENVIRONMENT = os.getenv('DJANGO_ENV', 'dev')

if ENVIRONMENT == 'prod':
	os.environ.setdefault(
		'DJANGO_SETTINGS_MODULE',
		'config.settings.prod'
	)
else:
	os.environ.setdefault(
		'DJANGO_SETTINGS_MODULE',
		'config.settings.dev'
	)


This ensures:
	- Local -> dev settings
	- Server -> prod setting


Update .env
~~~~~~~~~~~
Now control environment here:
DJANGO_ENV = dev

For production:
DJANGO_ENV = prod
